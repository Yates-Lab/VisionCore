## Why we're doing this

The `methods_eyepos_matching/` folder developed two estimator extensions on top
of McFarland's cross-trial LOTC decomposition (Extension 1: consistent time-bin
weighting under variable $n_t$; Extension 2: eye-position-distribution matching
under a non-homogeneous stimulus). We are now about to **propagate this new
pipeline into production** — replacing `VisionCore/covariance.py` and the
`ryan/fig2/` + `ryan/fig4/` (and `ryan/fig3/`) data pipelines with the
improved estimator from this folder.

**Before touching production**, we must vet the new pipeline against the current
production pipeline panel-by-panel. The contract:

1. **The new pipeline run with `target='naive'` must reproduce the current
   production results** (within numerical tolerance). If it doesn't, the new
   pipeline has changed something other than the distribution matching, and we
   need to find and explain every such difference before trusting the `full`
   numbers.
2. **The new pipeline run with the new default `target='full'`** (the actual
   viewing distribution $p(e)$) is the *intended* replacement. We want to see,
   for every panel, exactly how the headline numbers move from naive→full.

So for **each panel** we produce a **3-up side-by-side**:

- **(1) PROD** — the panel exactly as currently generated by `ryan/fig2` /
  `ryan/fig3` from `compute_fig2_data.load_fig2_data()`.
- **(2) NAIVE** — the same panel regenerated from the methods pipeline with
  `target='naive'`. **This must match (1).**
- **(3) FULL** — the same panel from the methods pipeline with `target='full'`
  (new default).

## Panels in scope

| panel | file | what it shows |
|---|---|---|
| fig2 **C** | `ryan/fig2/generate_fig2c.py` | per-subject histogram of FEM fraction $1-\alpha$ at the primary window |
| fig2 **E** | `ryan/fig2/generate_fig2e.py` | population **Fano factor** (slope-through-origin) vs counting window, uncorrected vs FEM-corrected, clustered-bootstrap CIs |
| fig3 **B** | `ryan/fig3/generate_fig3b.py` | noise-correlation scatter (corrected vs uncorrected) at primary window |
| fig3 **C** | `ryan/fig3/generate_fig3c.py` | mean Fisher-$z$ noise correlation vs window, uncorrected vs corrected |
| fig3 **D** | `ryan/fig3/generate_fig3d.py` | $\Delta z$ (corrected − uncorrected) vs window against the shuffle-null 95% band |

All five panel functions take `plot_panel_X(ax=None, refresh=False, data=None)`
and read everything they need out of the `data` dict returned by
`ryan/fig2/compute_fig2_data.load_fig2_data()`. **This is the seam to exploit**:
if we can produce a `data` dict in that same schema from the methods pipeline
(per target), the *existing* panel functions render the naive and full versions
with zero plotting-code divergence — any difference is then guaranteed to be in
the data, not the drawing.

## Known discrepancies to run to ground (don't assume, verify)

1. **Fano factor too high.** At the 25 ms window the production *corrected*
   population Fano was **< 0.8**; the new pipeline currently reports ≈ 0.9+.
   Suspects, in priority order:
   - **statistic mismatch** — production reports geomean / slope-through-origin
     over `positive=True`-paired units; ad-hoc per-cell medians are NOT
     comparable. Always compare the *same* statistic.
   - **inclusion mismatch** — see (2): clipping vs excluding uninterpretable
     units changes which cells enter the Fano.
   - **Crate estimator** — `decompose_trajectory` returns a *centred* Crate;
     `pipeline.decompose_session` overrides with the legacy *uncentred*
     `MM − Erate⊗Erate`. Confirm the methods Fano uses the uncentred Crate
     (it should, via `decompose_session`).
   - **count vs trajectory window** — the Fano is a count-per-mean and scales
     with the *count* window (`t_count`); `1-alpha` is a ratio and is
     window-stable. Make sure naive comparisons hold `t_count` fixed to the
     production `WINDOW_BINS` and **`t_hist≈1`** (NOT the §4.5 demo config
     `t_hist_ms=92`, which `generate_realdata.py` uses for a different purpose).
2. **Way more units with `1-alpha` clipped to 0.** Production keeps `alpha`
   **raw** and (per our prior convention) **excludes uninterpretable units with
   `1-alpha < 0` or `> 1`**; the new estimator **clips** to [0,1] instead, so
   those units survive and pile up at 0/1. **Find exactly where the old
   convention drops them** (panel code? a filter in `_metrics_one` /
   `_compute_*`? `paired_valid`?) and replicate it: add an option to the methods
   side to *exclude* (not clip) `1-alpha ∉ [0,1]` so the naive comparison is
   apples-to-apples. Quantify how many units this is, per window, per subject.
3. **Noise correlations not yet checked.** §4.5 deliberately deferred NC. The
   methods `nc_stats` exist but have never been validated against production
   (fig3 B/C/D). Also note the **shuffle null** (panel D) is currently produced
   only for `target='naive'` (`Shuffled_Crates` is computed for naive in
   `decompose_session`); decide whether `full` needs its own shuffle null for a
   fair panel-D "full" version, or whether the naive null is the right reference
   band.

## Phase 1 — step-by-step pipeline diff (do this FIRST, write it up)

Before any plotting, produce a written, itemized comparison of the two pipelines
and pin down every place results could diverge. At minimum:

- **Sessions/subjects**: same session list? `aligned_sessions.pkl` is Allen+Logan
  (25 sessions); production `SUBJECTS` includes Luke — is Luke present in the
  data or dropped? Confirm identical session sets enter both.
- **Unit inclusion**: rate/psth_r2/contam thresholds, `MIN_VAR`, finiteness, the
  `valid.sum()<3` session drop, and the `1-alpha∈[0,1]` exclude-vs-clip question.
  Produce a per-session table: `n_units_prod_valid` vs `n_units_methods_valid`
  and the set difference, with the reason each differing unit is in/out.
- **Counting windows**: confirm bins↔ms map identically (`[1,2,3,6]` @ `1/120`).
- **Estimator**: production single-bin `run_covariance_decomposition`
  (`intercept_mode`) vs methods trajectory-mode at `t_hist≈1`; Cpsth
  (`mcfarland` vs `split_half`); Crate centred vs uncentred; Ctotal weighted vs
  unweighted (`_ctotal_unweighted`).
- **Statistics**: Fano (geomean + slope-through-origin, `positive=True` pairing),
  NC (PSD projection, Fisher-$z$, per-session clustering), shuffle nulls
  (seeds, n_shuffles). These must be byte-for-byte the same function on both
  sides — ideally *literally* the production functions.

Deliver Phase 1 as the first section of `note_consistency.md` with a clear table
of every identified difference and whether it is expected (distribution change)
or a bug to fix before proceeding.
